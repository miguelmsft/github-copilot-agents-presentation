Example: Security Policy Enforcement

.github/hooks/scripts/security-check.sh:

#!/bin/bash
set -euo pipefail

INPUT="$(cat)"
TOOL_NAME="$(echo "$INPUT" | jq -r &#x27;.toolName')"
TOOL_ARGS="$(echo "$INPUT" | jq -r &#x27;.toolArgs')"

# Only check bash commands
if [ "$TOOL_NAME" != "bash" ]; then exit 0; fi

COMMAND="$(echo "$TOOL_ARGS" | jq -r &#x27;.command')"

# Block privilege escalation
if echo "$COMMAND" | grep -qE &#x27;\b(sudo|su)\b'; then
  jq -n &#x27;{
    permissionDecision: "deny",
    permissionDecisionReason: "Privilege escalation blocked."
  }&#x27;
  exit 0
fi
Here's a real security hook. It reads JSON from stdin, extracts the tool name and arguments, and checks if the agent is trying to run a dangerous command. If it detects `sudo` or `su`, it returns a deny decision with a reason. The agent sees this reason and adjusts — maybe it asks you to run the command manually. You can block anything: `rm -rf`, `curl | bash`, database drops, whatever your security policy requires. Quick terminology note: the field names like `toolName` and `toolArgs` use a naming convention called "camelCase" — words are joined together and each word after the first starts with a capital letter. This is common in JavaScript and JSON.
